# Epic 1 — Function-Correctness & Export-Parity — Design

> Part of the Lost Media Emulator v2 roadmap (`docs/superpowers/specs/2026-06-27-v2-roadmap-design.md`),
> Track Q, sequenced after Epic 0 (reference audit, shipped). This epic gets its own implementation plan.

## Goal

Two guarantees for v2:

1. **Every feature actually works** — no dormant or half-wired controls (a slider/toggle that
   renders in the UI but changes nothing, or only works in preview, or has a broken/missing state).
2. **What you see is what you get** — the export matches the preview **at any frame index**,
   including animated/temporal effects. This is the failure class that produced the
   export-resolution bug and the OSD-export desync.

Audit-style and tooled, reusing the existing `src/lib/export-validator.js`
(`validateExportAgainstPreview`), which already implements the two core checks:
- **Determinism** — a byte-identical re-render of the export path (passA vs passB). The hard pass.
- **Preview↔export parity** — the export-path CPU render vs the displayed preview pixels, accepted
  within a tolerance (Δmean ≤ 6 for a CPU preview, ≤ 12 for a GPU preview). The soft pass.

## Non-goals (out of scope for this epic)

- Fixing the *authenticity* of effects — that is Epic 3, driven by `docs/audit/FIX-LIST.md`. Epic 1
  only cares that an effect renders **identically in export and preview**, not whether it looks
  period-correct.
- The OSD rebuild (Epic 2, already spec'd in `docs/OSD-REBUILD-HANDOFF.md`). Epic 1 will *detect*
  OSD parity gaps and fix-list them; the structural rebuild is Epic 2's job.
- GPU engine leap (Epic 6). Epic 1 validates the current CPU pipeline.

## Architecture — two phases

### Phase 1 — Frame-accurate parity sweep (automated)

A sweep harness, a sibling of `tools/gpu-coverage.snippet.js` / the audit contact-sheet, run in the
app's browser context (the renderer and `export-validator` are canvas/browser code, so the harness
runs there — driven via the preview tooling or pasted into devtools, the same pattern that worked
for the Epic 0 audit). A heavier headless/vite-node harness was rejected because it fights the
canvas dependency.

For each of the 91 presets, at **sampled frame indices** chosen to exercise temporal effects —
`t = 0` plus `1, 7, 15, 29` (covers line-jitter parity, dropout/flicker animation, head-switching,
wow/flutter drift, frame-stutter, and the OSD timestamp tick) — the harness:

1. Renders the **preview** path frame (the live renderer at frame t).
2. Renders the **export** path frame (the export-path CPU render at the same frame t — the exact
   code path export uses, via `validateExportAgainstPreview`).
3. Records the two existing checks: determinism (passA vs passB byte-identical) and parity
   (export vs preview, Δmean vs tolerance).

**Two layers, kept honestly separate:**

- **(a) Renderer-level parity** — export-path render == preview render at frame t. Fully
  automatable in-browser; this is the bulk of the sweep and where the temporal class lives.
- **(b) Encode-level parity** — the decoded MP4 frame == the renderer output (the YUV conversion /
  chroma-subsampling / scaling shift introduced by the ffmpeg encode). Full-91 encoding is too slow
  to sweep, so this layer is **spot-checked** by extending the existing electron ffmpeg e2e smoke
  tests (`electron/__tests__/ffmpeg-pipeline.smoke.test.js`) on a small representative set
  (e.g. one clean, one heavy-temporal, one ProRes), ffprobe/decode-comparing a frame.

**Output:**
- `docs/parity/PARITY-FIX-LIST.md` — every preset×frame mismatch, ranked by severity (how far over
  tolerance, determinism failures first), with the suspected cause and a fix lane.
- `docs/parity/coverage.md` — a living tracker: presets×frames swept, pass/fail counts, % parity-clean
  (mirrors the audit's `coverage.md`, regenerated by the harness).

### Phase 2 — Deep feature-correctness pass (manual)

What pixel-diffing cannot catch: a control that does nothing, a feature wired only into preview,
a missing/broken empty/disabled/loading state.

- **Inventory** every interactive control, toggle, and panel across the surfaces (the ~30
  components + the top bar + status bar + Export dialog + command palette + Masks painter + audio).
- For each, verify three things: (1) it actually changes the render or app state (not a dead
  control); (2) its effect survives export (cross-checked against the Phase 1 results); (3) its
  enabled / disabled / empty / loading states are correct and explained (e.g. the gated "Master"
  export chip should say why it's disabled).
- **Output:** `docs/parity/FEATURE-CHECKLIST.md` — status per feature (works / dead / half-wired /
  state-gap) — then fix the dead and half-wired ones.

## Components & layout

- `tools/parity/parity-sweep.snippet.js` — the in-app sweep harness (Phase 1).
- `src/lib/parity/` — any *new* pure helpers needed beyond `export-validator` (e.g. severity
  ranking, frame-sampling selection), TDD'd. Reuse `export-validator.js` for the actual pixel
  compare; do not duplicate its logic.
- `docs/parity/` — `PARITY-FIX-LIST.md`, `coverage.md`, `FEATURE-CHECKLIST.md`, `README.md`
  (documents the repeatable sweep command + the per-frame method, like the audit README).
- Extend `electron/__tests__/ffmpeg-pipeline.smoke.test.js` for the encode-level spot-check.

## Error handling & edge cases

- A preset that throws during render is recorded as a hard failure in the sweep (not skipped silently).
- Temporal effects seeded by frame index must be deterministic given (params, frameIndex) — if a
  preset uses wall-clock or `Math.random` without a seed, that surfaces as a determinism failure
  (passA ≠ passB) and is itself a fix-list item.
- GPU vs CPU preview mode changes the parity tolerance (12 vs 6) — the harness records which mode
  each result was taken under, and sweeps the **CPU preview** path as the authoritative baseline
  (tightest tolerance), noting GPU-mode deltas separately.

## Testing

- New `src/lib/parity/` helpers are TDD'd (vitest), keeping the existing 96 tests green and `tsc`
  clean.
- The encode-level spot-check rides the existing ffmpeg smoke-test harness.
- The sweep harness itself is a tool (visual/operational), validated by running it and confirming
  the coverage numbers, not unit-tested.

## Done criteria (Epic 1 complete for v2)

1. Every preset passes frame-accurate **renderer-level** parity at the sampled frames, OR is in
   `PARITY-FIX-LIST.md` with the must-fix (determinism failures + large parity deltas) items landed.
2. Encode-level spot-checks pass on the representative set.
3. `FEATURE-CHECKLIST.md` complete; no dead or half-wired controls remain; state-gaps fixed.
4. 96 existing tests green + new parity helper tests; `tsc` clean.
5. The sweep is a documented, repeatable command (so it can guard future changes + the Epic 6 port).

## Relationship to other epics

- Feeds **Epic 3** (effect quality): parity must hold before authenticity fixes, so effects aren't
  fixed twice (once in preview, again to re-match export).
- Feeds **Epic 6** (GPU engine leap): the parity sweep becomes the pass/fail harness for porting
  each effect to the shader core against the same bar.
- Surfaces (but does not fix) OSD parity gaps for **Epic 2**.
